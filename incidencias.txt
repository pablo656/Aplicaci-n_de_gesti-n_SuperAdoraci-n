3.3.1. Definir un protocolo para resolución de incidencias

Durante el desarrollo y la puesta en marcha de la aplicación web SuperAdoracion
se establece el siguiente protocolo para gestionar cualquier incidencia que
pueda surgir, tanto en fase de desarrollo como en producción.

--------------------------------------------------------------------------------

3.3.1.1. Recopilación de información

Ante cualquier incidencia detectada, se recopilará la siguiente información:

  - Fecha y hora exacta en que se detectó el problema.
  - Usuario o rol afectado (cliente, administrador, dueño).
  - Descripción detallada del error: mensaje de error mostrado, código HTTP
    si aplica (500, 403, 404...), comportamiento inesperado observado.
  - Pasos exactos para reproducir el error.
  - Capturas de pantalla o registro de errores del servidor (error_log de
    Apache/PHP en XAMPP: C:\xampp\apache\logs\error.log).
  - Entorno: navegador utilizado, sistema operativo, versión de XAMPP.
  - Módulo o sección de la aplicación afectada (Log in, Sign in, catálogo
    de productos, reservas, panel de administración, etc.).

--------------------------------------------------------------------------------

3.3.1.2. Posible solución

Una vez recopilada la información, se sigue el siguiente proceso de resolución:

  1. Análisis de la causa raíz: revisar el código del modelo, controlador o
     vista afectados (arquitectura MVC) y consultar el log de errores de PHP.

  2. Reproducción del error en entorno local (XAMPP) antes de aplicar
     ningún cambio en producción.

  3. Corrección del código: se aplica el fix en la rama 'developer' del
     repositorio Git, se prueba en local y, una vez validado, se fusiona
     con la rama 'main'.

  4. Verificación: se comprueba que el error ya no se reproduce y que el
     resto de funcionalidades relacionadas siguen operando correctamente.

  Incidencias habituales identificadas durante el desarrollo:

  - Error en formularios de Log in / Sign in: validación incorrecta de
    credenciales y gestión de sesiones ($_SESSION). Solución: revisión del
    controlador de autenticación y corrección de la lógica de comprobación
    en la base de datos (tabla usuarios).

  - Error de stock negativo: triggers de base de datos (trg_reserva_insert,
    trg_reserva_delete) no actualizando correctamente el stock. Solución:
    revisión y corrección de los triggers en el fichero supermercado.sql.

  - Imágenes de productos no visibles tras subida: ruta incorrecta al
    directorio mvc/imagenes/. Solución: corrección de la ruta relativa en
    el controlador de productos.

  - Pérdida de sesión inesperada: session_start() no llamado antes de
    acceder a $_SESSION en alguna vista. Solución: asegurar session_start()
    al inicio de cada Index*.php.

--------------------------------------------------------------------------------

3.3.1.3. Registro

Cada incidencia resuelta se registra con la siguiente ficha:

  +----------------------+----------------------------------------------------+
  | Campo                | Descripción                                        |
  +----------------------+----------------------------------------------------+
  | ID Incidencia        | Identificador único (ej: INC-001)                  |
  | Fecha                | Fecha y hora de detección                          |
  | Prioridad            | Alta / Media / Baja                                |
  | Módulo afectado      | Sección de la app donde se produjo el error        |
  | Descripción          | Explicación del error detectado                    |
  | Causa raíz           | Motivo técnico del fallo                           |
  | Solución aplicada    | Cambios realizados para resolverlo                 |
  | Responsable          | Desarrollador que resolvió la incidencia           |
  | Commit Git           | Hash del commit con la corrección aplicada         |

  | Descripción          | Pérdida de integridad de datos: el stock de        |
  |                      | productos en la aplicación no coincidía con el     |

  |                      | actualizaba automáticamente, provocando que los    |

  |                      |    del stock del producto automáticamente.         |
  |                      | 2. Creación del trigger trg_reserva_delete:        |
  |                      |    al eliminar una reserva devuelve la cantidad    |
  |                      |    al stock del producto automáticamente.          |
  |                      | 3. Creación del evento evt_borrar_reservas:        |
  |                      |    elimina reservas con más de 7 días cada día.    |
  |                      | 4. Creación del evento evt_borrar_pedidos:         |
  |                      |    elimina pedidos con más de 30 días cada día.    |

  | Descripción          | La aplicación no limita el número de intentos      |
  |                      | fallidos de inicio de sesión. Un atacante puede    |
  |                      | realizar un ataque de fuerza bruta contra          |

  |                      | tabla usuarios en la BD.                           |
  |                      | En el controlador de autenticación:                |
  |                      | - Si las credenciales son incorrectas,             |
  |                      |   incrementar intentos_fallidos en 1.              |
  |                      | - Si intentos_fallidos >= 8, activar               |
  |                      |   bloqueado = 1 e impedir el acceso.               |
  |                      | - Si el login es correcto, resetear                |
  |                      |   intentos_fallidos a 0.                           |
  |                      | - El dueño/administrador puede desbloquear         |
  |                      |   la cuenta desde el panel de gestión.             |
